## 架构变化

### 1. 追踪模型（A+B 混合）

每个 Issue 同时携带两套追踪数据：

**A. Lineage Chain（血缘链）** — 扁平数组，一眼看到完整上下游：

```json
{
  "lineage": [
    {"goalId": "g-001", "projectName": "ui-design", "issueId": "iss-101", "opcNode": "design-team", "label": "登录页设计稿"},
    {"goalId": "g-001", "projectName": "api-spec", "issueId": "iss-201", "opcNode": "backend-team", "label": "API 接口文档"},
    {"goalId": "g-001", "projectName": "frontend-dev", "issueId": "iss-301", "opcNode": "frontend-team", "label": "前端登录页实现"}
  ]
}
```

**B. Distributed Trace（OpenTelemetry 风格）** — 结构化追踪，对接 Jaeger：

```
traceId:    "g-001"                         (= root goal ID)
spanId:     "iss-301"                       (= 本 issue 唯一 ID)
parentSpans: ["iss-101", "iss-201"]         (= 上游 issue IDs)
```

### 2. 数据流

```
Master OPC (:9527)
  │
  ├─ createFederatedGoal        ← OTel span: "createFederatedGoal"
  │   ├─ DAG 拓扑排序
  │   ├─ Layer 0: dispatch      ← 无依赖的 project 并行下发
  │   │   └─ payload 携带 lineage=[]
  │   │
  │   └─ federationCallback     ← OTel span: "federationCallback"
  │       ├─ 更新 project 状态
  │       ├─ 注入上游 result 到下游 context
  │       ├─ Layer 1: dispatch  ← 依赖满足后下发，lineage 追加上游 ref
  │       └─ ...直到所有 layer 完成
  │
Remote OPC (:9528/9529/9530)
  │
  └─ runTask                    ← OTel span: "runTask"
      ├─ task.GoalID = 来自上游
      ├─ task.ProjectID = 来自上游
      ├─ task.LineageJSON = 来自上游
      ├─ 执行 agent
      └─ callback → Master（携带 result + lineage）
```

### 3. 新增字段

**IssueRecord（api/v1/types.go）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `GoalID` | string | 所属 Goal ID |
| `TraceID` | string | = root goal ID（OTel 兼容） |
| `SpanID` | string | 本 issue 唯一标识 |
| `ParentSpans` | []string | 上游 issue spanId 列表 |
| `LineageJSON` | string | JSON 序列化的 `[]LineageRef` |

**TaskRecord（api/v1/types.go）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `LineageJSON` | string | 从上游传递的血缘链 |

**AuditEvent（pkg/audit/audit.go）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `IssueRef` | string | 关联 issue 引用（补全审计追踪链） |

## 新增 CLI Flags

### `opctl serve`

| Flag | 默认值 | 说明 |
|------|--------|------|
| `--log-level` | `info`（来自 config） | 日志级别：`debug\|info\|warn\|error` |
| `--state-dir` | `~/.opc/state` | 状态目录（支持多实例隔离） |
| `--otel` | `false` | 启用 OpenTelemetry 追踪 |
| `--otel-endpoint` | `localhost:4318` | OTLP HTTP 端点 |
| `--otel-service` | `opc-{port}` | OTel 服务名 |

### config.yaml 配置

```yaml
logLevel: debug    # debug | info | warn | error
```

## 多联邦测试流程

### 前置条件

```bash
# 1. 重新编译 opctl（包含新 flags）
go build -o opctl ./cmd/opctl
sudo cp opctl /usr/local/bin/opctl   # 或加入 PATH

# 2. 启动 Jaeger
docker run -d --name jaeger \
  -p 16686:16686 \
  -p 4318:4318 \
  -e COLLECTOR_OTLP_ENABLED=true \
  jaegertracing/all-in-one:1.54
```

### 方式一：脚本启动（本地多实例）

```bash
cd examples/federation-workflow

# Step 1: 启动 4 个 OPC 实例（各自隔离的 state-dir）
bash start-federation.sh

# Step 2: 注册联邦关系
bash register-companies.sh

# Step 3: 下发带依赖的 Goal
bash dispatch-goal.sh

# Step 4: 观察
# Jaeger UI — 全链路追踪
open http://localhost:16686

# 查看各节点任务
curl http://localhost:9527/api/goals | jq          # Master
curl http://localhost:9528/api/tasks | jq          # Design
curl http://localhost:9529/api/tasks | jq          # Frontend
curl http://localhost:9530/api/tasks | jq          # Backend

# 查看某个任务的血缘链
curl http://localhost:9529/api/tasks/<taskId> | jq '.lineage'

# Step 5: 停止
bash stop-federation.sh
docker stop jaeger && docker rm jaeger
```

### 方式二：Docker Compose 一键启动

```bash
# 启动 Jaeger + 4 OPC 节点
docker compose -f docker/docker-compose.observability.yaml up -d

# 停止
docker compose -f docker/docker-compose.observability.yaml down
```

### 方式三：手动逐步测试

```bash
# 1. 启动 Master
opctl serve --port 9527 --otel --log-level debug --state-dir /tmp/opc-master

# 2. 启动 Design 节点
opctl serve --port 9528 --otel --otel-service opc-design --state-dir /tmp/opc-design

# 3. 注册联邦
curl -X POST http://localhost:9527/api/federation/companies \
  -H "Content-Type: application/json" \
  -d '{"name":"design-team","endpoint":"http://localhost:9528","type":"software","agents":["coder"]}'

# 4. 下发简单 Goal（无依赖，直接执行）
curl -X POST http://localhost:9527/api/federation/goals \
  -H "Content-Type: application/json" \
  -d '{
    "name": "test-goal",
    "description": "测试联邦下发",
    "companies": ["<company-id>"]
  }'

# 5. 下发带依赖的 Goal
curl -X POST http://localhost:9527/api/federation/goals \
  -H "Content-Type: application/json" \
  -d '{
    "name": "带依赖的 Goal",
    "description": "测试依赖编排",
    "projects": [
      {"name": "step-a", "companyId": "<id-1>", "agent": "coder", "description": "第一步"},
      {"name": "step-b", "companyId": "<id-1>", "agent": "coder", "description": "第二步", "dependencies": ["step-a"]}
    ]
  }'
```
